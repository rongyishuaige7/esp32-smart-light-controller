# 安全边界与漏洞报告

这是一个基于 ESP32、四路**低压 LED**和可选 BH1750 的本地教学原型，不是智能家居成品、网络安全设备、可靠远程控制系统、电气安全产品或照明控制器。

## 网络边界

- 公开默认固件没有 Wi‑Fi 凭据，不创建接入点，也不启动 HTTP；只有本机创建 Git 忽略的 `wifi_credentials.h` 并成功连接可信局域网后，HTTP API 才会启动；
- 可选 HTTP API 没有 TLS、身份认证、权限模型、会话、请求签名、速率限制、审计或设备身份校验；
- 浏览器页面和 API 只使用同源请求，公开整理已移除宽松 `Access-Control-Allow-Origin: *`；这**不等于**接口已认证或可安全暴露；
- 仅可在隔离、可信的本地网络中使用；不得暴露到公网、端口转发、共享热点、不可信 Wi‑Fi 或多租户网络；
- `POST /api/led` 只能表示固件接受一次低压 LED 状态请求，不代表物理 LED 已改变、网络可达、控制必达、有人处理或持续可用；
- 公开版不提供 HTTP 网络配置或重置接口。若需要改变网络凭据，只能在受控的本机文件维护过程中执行，并先确认保存数据与操作后果；
- `wifi_credentials.h` 和网络环境可能包含 Wi‑Fi 信息；实际 Arduino/ESP32 Wi‑Fi 栈或你的维护操作也可能把网络信息写入设备持久化区域。不要公开本机凭据文件、NVS 导出、串口日志、浏览器截图、Wi‑Fi 名称、密码、私网 IP、MAC、网络拓扑或二维码。

## 硬件与功能边界

- 仅使用低压 LED、受控电源和经过核对的限流/驱动电路；不得连接市电、公共照明、应急照明、消防、安防、门禁、医疗、生产控制或无人值守场景；
- 本固件默认避免开机四路全亮测试，但真实复位/上电的电气行为仍由实际电路决定；
- BH1750 不可用或读数失败时，状态 API 返回 `sensor: "unavailable"` 与 `lux: null`，自动模式不应根据无效读数改变输出；
- 自动模式会同时覆盖四路输出并清除各路倒计时，不是逐路独立智能控制；
- 阈值、迟滞、倒计时和 HTTP 成功响应不构成节能、照度、安全、可靠性、长期稳定性或控制送达承诺。

## 当前验证边界

当前公开候选尚未按公开 commit 重新烧录或联调 ESP32、四路 LED、BH1750、NVS、本机凭据连接、局域网 HTTP、倒计时或自动模式。请先阅读 [HARDWARE.md](HARDWARE.md)、[docs/PROJECT_STATUS.md](docs/PROJECT_STATUS.md) 和 [docs/VERIFICATION.md](docs/VERIFICATION.md)。构建和 CI 不能证明任何真机、电气或网络行为。

## 报告方式

不要在公开 Issue 中提交 Wi‑Fi 凭据、私网信息、NVS/Flash 导出、MAC、完整网络拓扑或可复用攻击细节。请通过 GitHub 账户主页的公开联系方式提供最小化、已脱敏的复现说明；在收到确认前，不要公开可能影响其他复现者的细节。
